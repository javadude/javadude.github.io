Assignment 2 - Contact Manager

In this assignment, you'll create a simple contacts application. You'll work with the following concepts:

  • GUI Components: Activities, TextViews, EditTexts, Layouts, Lists, Fragments
  • Data Storage: Databases and Content Providers
THIS ASSIGNMENT WILL TAKE SOME TIME TO IMPLEMENT. READ THROUGH AND MAKE SURE YOU UNDERSTAND EVERYTHING REQUIRED AS SOON AS POSSIBLE. DO NOT WAIT UNTIL THE WEEK BEFORE THE ASSIGNMENT IS DUE TO START!!!

I recommend the following strategy:

  1. Create your activities and fragments with just the views and no data management. Note that in the class videos, I describe converting from a strictly activity-based example to a fragment-based example. You don't need to do this (though you can if you would like); you can immediately start with fragments.
  2. Ensure that navigation between the fragments works properly. You may want to set up a dummy ArrayAdapter at first to test simple clicks on the list for navigation.
  3. Create your contact model object and associate it with the fields in each fragment. For starters, you may want to hardcode a sample contact instance. If you would like, you can make the Contact Parcelable at first (but be sure to remove the Parcelable support before turning in the assignment) and pass contacts between activities.
  4. Create your content provider
  5. Set up database reads and writes - have each fragment lookup the contacts independently; you should only pass ids to them.

Note that I recently combined two assignments into this single assignment; if you notice any typos or something seems to conflict, please let me know as soon as you see it.

Overview

For this assignment, you'll need to create:

  • A Contact class
    • id (long)
    • First and Last Name (separate Strings)
    • Birthday (Calendar)
    • Phone numbers (Strings): Home, Work, Mobile
    • Email Address (String)
  • Three Fragments
    • ListView of all contacts
    • Scrolling layout of views to display a contact's data
    • Scrolling layout of views to edit a contact's data. (Note: The birthday might be a TextView with a “…” button to edit its value)
  • Main “list” activity
    • If in portrait orientation (res/layout dir)
    • Displays just the list fragment
    • “new contact” action button starts “edit” activity to create a new contact
    • Tapping a contact starts the “display” activity with that contact displayed
    • If in landscape orientation (res/layout-land dir)
    • Displays “list” AND either the “display” or “edit” fragment
    • (Note - layout contains all three fragments, but you'll use the fragment manager to switch between them)
    • If no contact selected, you should only display the “list” fragment
    • Action bar icons will change based on state
      • New Contact action is always available
      • If “display” is shown, have edit action -> when pressed, swap to the “edit” fragment with the selected contact
      • If “edit” is shown, have done and cancel actions -> both swap back to the “display” fragment, but “done” saves the data before doing so
    • Tapping a contact shows the “display” fragment for the selected contact
  • “Display” activity
    • Displays just the display fragment
    • Has “edit” button on action bar -> sends user to “Edit” activity
  • “Edit” activity
    • Displays just the edit fragment
    • Has “done” and “cancel” buttons on action bar. Both finish() the activity, but the “done” button saves the changed data first
  • Content provider to manage the contact data

Note that there is no “delete” function. To do this right, you should really have a Contextual Action Bar (to allow selecting and deleting multiple items from the list), which isn't difficult, but is “one more thing” to implement, so I omitted it from your requirements.

Also note that this assignment isn't about making a pretty user interface; it's really about the flow of applications like this, using data bases, and using fragments in a GUI.

General Notes

Be sure to name this project and its packages starting with lastname.firstname.hw2.

When defining your list adapter's getView() method, use the following call to create a view (when needed)

convertView = getLayoutInflater().inflate(R.id.your_row_layout, parent, false);

This version of the inflate method creates the view using the parent for layout context, but doesn't actually attach it to the view (the “false” parameter - this is very important, as the ListView manages the add/remove of the view.)

You must define your own custom layout for the row; do not use any of the ones that android provides (such as android.R.id.simple_list_item1).

Generate icons in Eclipse using “New->Android Icon Set”. You must use the following icons. Note that you may need to change to Holo Light or Holo Dark depending on which layout you selected. (If you used Holo Light with Dark Action bar, use Holo Dark for the icons)


New Contact
Edit
Done
Cancel

You will define and handle your action bar menus in the fragments, not the activities. Note that you will need to call setHasOptionsMenu(true); in onCreateView as well as define onCreateOptionsMenu.

In your list view show the full name of the contact and one of the phone numbers for it. You can design the row layout however you would like, one line or two per contact.

For your list adapter, use a CursorAdapter as described in Module 7, mapping the database fields to your custom row layout. Note that to do this, you'll need separate TextViews in your layout for the first and last names. Note: if you would like to try to get the app running before you have the database code set up, you can create a subclass of BaseAdapter to display some dummy objects in the list.

Your display and edit fragments should scroll (there's likely not enough vertical room for all of the fields)

Choose appropriate inputTypes for the fields (phone numbers, email, etc)

When setting up your intents to communicate between activities, use at least one explicit intent and at least one implicit intent. (Note that this only applies when the application is in portrait mode; in landscape mode you'll never switch activities; you'll just pass data between fragments)

Be creative on how you lay out your activities, but make sure they look 'reasonable'.

Hardcoded data (such as field labels) must not appear in the xml layout or Java classes unless I ok it in class; you must use the strings.xml file. (This note will not be repeated in later assignments but still applies).

Content Provider Notes

You do not need to set up the code to perform database upgrades! This assignment acts as version 1 of your database. (If you're following the example code, be sure to remove the extra work done during database update…)

Create a content provider to manager your contacts, similar to the 'todo' content provider demonstrated in class.

When creating your database, be sure to name your id field _id as described in class (integer primary key autoincrement)

As described above, use a CursorAdapter as described in Module 7 to display your data from the database in the list view.

Pass the id of the contact to edit or display in your Intent used to start the “display” or “edit” activities. If you're creating a new Contact, pass -1 as the id. The receiving activity will look up the requested contact by that id (or create a new one if it were -1)

On the edit activity, read the contact from the database (if the id passed to it was not -1). When the done button is pressed, either insert a new contact in the data base (if creating a contact) or update an existing contact. Do not return the contact to the caller.

In activities that need to update information, re-read the contact or contact list. The easiest way to do this is to put the read-from-database-and-insert-into-fields logic into your onResume() method; this allows you to use the exact same code whether you're entering the activity for the first time or returning to it.

Be sure the only way that contacts are stored/retrieved is through the contact provider (using getContentManager() or the CursorLoader)

When storing the birthday in the database, you can store it as a single long value, individual year/month/day integers, or a string representation that you parse. Note that you cannot simply stash a Calendar or Date object in the database

Fragment Notes

Do not worry about dealing with screen rotation while the date picker dialog is visible. This is very tricky to properly implement with this application design and we will not be testing it for grading. I'm working on some frameworks to make this simpler.

Your activities should focus on navigation; the fragments should focus on managing the data in the UI. You should not have any calls to startActivity in your fragments, nor should you lookup or manage views in the activities.

When in portrait orientation, the list/display/edit of a contact should appear in separate activities. When in landscape orientation, the list/display/edit of a contact should appear side by side in the same activity. (Note: in real life, you'd probably only do this on a device with a larger screen, like a tablet, but for this example, do it on a phone. Note that it may appear a bit cramped; try to create a reasonable looking user interface

Use the DatePickerDialogFragment discussed in class to edit the birthday.
