What's the problem?

Suppose you write a little application with code like the following:

Figure 1: Bad I/O example

// THIS IS EVIL CODE -- DO NOT DO THIS!!!

try {

    FileReader r = new FileReader("somefile.txt");

    // process r

    r.close();

} catch (IOException e) {

    // report and/or log error

}

Sure, it looks reasonable at first glance, but then so did COBOL.

The first, and possibly most obvious problem is the position of the close() call. If anything goes wrong, throwing an exception between the open and close of the file, the close() call will be skipped! This has the potential of leaving file handles allocated in the OS.

Classes in the Java libraries such as FileReader have a finalize() method that will close() the system resource when the FileReader is garbage collected. Unfortunately, there is no way to determine when this will occur, and if the FileReader object is held for a long segment of code it may delay the finalize() even further. In a multi-user system, this can quickly cause resource allocation issues. Even in a single-use system, there could be multiple applications in contention for resource handles. It's critical that these resources be freed as soon as you're finished using them.

Using finally to ensure we try to close

The first, and most important change, is to move the close() call into a finally block. Java's finally block is always executed:

  • If the try completes successfully, finally is run after it.
  • If an exception is thrown in the try block and caught by a catch block, the finally runs after the catch block.
  • If an exception is thrown in the try block and not caught by a catch block, the finally runs immediately before passing the exception on to the containing block or calling method.

Moving the close() call to the finally block looks as follows:

Figure 2: Getting better...

// THIS IS EVIL CODE -- DO NOT DO THIS!!!

FileReader r = null;

try {

    r = new FileReader("somefile.txt");

    // read from r

} catch (IOException e) {

    // report and/or log error

} finally {

    if (r != null)

        try {

            r.close();

        } catch (IOException e) {

            // report and/or log error

        }

}

Isn't that pretty? We're now set up so that no matter what happens in the outer try block, we'll at least attempt to close the file (assuming it was opened successfully; that's what the null check is for).

But unfortunately, this still isn't quite right.

Pending exceptions

Java only keeps track of a single exception being thrown. What happens if an exception is thrown during the file read and during the file close? The last exception wins. If an exception is thrown in both places, the caller will be notified of the close error, and not the read error.

This is not a good situation; we're probably more interested in the exception that was thrown in the try block than we are the one thrown during the close. So let's modify things a bit more.

Figure 3: Keeping track of the pending exception

        FileReader r = null;

/* 1 */ Throwable pending = null;

        try {

            r = new FileReader("somefile.txt");

            // read from r

/* 3 */ } catch (ThreadDeath t) {

/* 3 */     throw t;

/* 2 */ } catch (Throwable t) {

/* 2 */     pending = t;

        } finally {

            if (r != null)

                try {

                    r.close();

                } catch (ThreadDeath t) {

                    throw t;

                } catch (Throwable t) {

                    // report and/or log error

/* 4 */             if (pending == null)

/* 4 */                 pending = t;

                }

/* 5 */     if (pending != null)

/* 5 */         if (pending instanceof SomeWrapperException)

/* 5 */             throw (SomeWrapperException) pending;

/* 5 */         else 

/* 5 */             throw new SomeWrapperException(pending);

        }

Wow. That's some fun code to noodle through. What's happening? (The numbers below map to the numbers in the comments in the above code)

  1. We need to keep track of the exception we care about and want to return to the caller.
  2. Because the pending exception could be any exception, we need to catch Throwable
  3. Catching Throwable is normally a bad idea; it includes ThreadDeath, which needs to be re-thrown for proper thread notification of impending thread doom and destruction. We set up a catch block just so we can rethrow it.
  4. If we get an exception during the close call, we check to see if we already are tracking a pending exception. If not (and only if not), we can track the close exception. If we're already tracking an exception, we should log the close error so we still know it happened.
  5. If we have a pending exception, we need to convert it to a type of exception that can exit the method. We can create a custom exception for this (checked or unchecked; depends on your preference), or wrap it in RuntimeException. If the pending exception is already the same type as the wrapper, we can just re-throw it.

Isn't that fun? Before we look into how to simplify it, let's make things a little worse

Suppose you have nested streams; perhaps you want to wrap an output stream in a buffer for better performance.

Figure 4: Nested stream handling

        FileReader r = null;

/* 1 */ BufferedReader br = null;

        Throwable pending = null;

        try {

            r = new FileReader("somefile.txt");

/* 1 */     br = new BufferedReader(r);

            // read from br

        } catch (ThreadDeath t) {

            throw t;

        } catch (Throwable t) {

            pending = t;

        } finally {

/* 2 */     if (br != null)

                try {

                    br.close();

                } catch (ThreadDeath t) {

                    throw t;

                } catch (Throwable t) {

                    // report and/or log error

                    if (pending == null)

                        pending = t;

                }

/* 3 */     if (r != null)

                try {

                    r.close();

                } catch (ThreadDeath t) {

                    throw t;

                } catch (Throwable t) {

                    // report and/or log error

                    if (pending == null)

                        pending = t;

                }

            if (pending != null)

                if (pending instanceof SomeWrapperException)

                    throw (SomeWrapperException) pending;

                else

                    throw new SomeWrapperException(pending);

        }

Crikey! That's a lot of code just to do a simple read from a file. Let's look at a few details in there:

  1. We declare and open the stream wrapper as we normally would (we declare it outside the try so we have access to it in the finally).
  2. We try to close the outermost wrapper first. If this succeeds, it will automatically try to close nested streams.
  3. We try to close the nested stream.

If a stream is already closed, further close() calls will be ignored (per the contract of Java's Closeable interface).

If the wrapping stream wasn't properly closed, it is important to try to close the nested stream explicitly to try to free up the underlying system resource.

We now have proper file closing!

But let's get real. How many folks are going to go this far? Even if they do, how many will get it right every time they write it?

Simplifying things

Let's find a way to factor out the necessary boilerplate so we can concentrate on our I/O design. We need to figure out which parts of this algorithm stay the same whenever we use it, and which parts differ.

Figure 5: Where are the variants?

// PSEUDO-CODE -- NOT JAVA!

// VARIANT: DECLARE THE STREAMS AND WRAPPERS

Throwable pending = null;

try {

    // VARIANT: OPEN AND PROCESS THE STREAMS

} catch (ThreadDeath t) {

    throw t;

} catch (Throwable t) {

    pending = t;

} finally {

    // VARIANT: FOR EACH STREAM (OR WRAPPER)

    if (STREAM != null)

        try {

            STREAM.close();

        } catch (ThreadDeath t) {

            throw t;

        } catch (Throwable t) {

            // report and/or log error

            if (pending == null)

                pending = t;

        }

    // END VARIANT: FOR EACH STREAM WRAPPER

    if (pending != null)

        if (pending instanceof SomeWrapperException)

            throw (SomeWrapperException) pending;

        else

            throw new SomeWrapperException(pending);

}

Hmm looks like a good bit of that doesn't vary. What if we use a List of Closeables rather than separate variables for each stream?

Figure 6: Collapsing the variants

// PSEUDO-CODE -- NOT JAVA!

List<Closeable> streams = new ArrayList<Closeable>();

Throwable pending = null;

try {

    // VARIANT: OPEN AND PROCESS THE STREAMS

    //     STORE EACH STREAM IN streams LIST

} catch (ThreadDeath t) {

    throw t;

} catch (Throwable t) {

    pending = t;

} finally {

    for (Closeable c : streams) {

        try {

            c.close();

        } catch (ThreadDeath t) {

            throw t;

        } catch (Throwable t) {

            // report and/or log error

            if (pending == null)

                pending = t;

        }

    }

    streams.clear();

    if (pending != null)

        if (pending instanceof SomeWrapperException)

            throw (SomeWrapperException) pending;

        else

            throw new SomeWrapperException(pending);

}

We're now down to a single spot that varies! Everything else is exactly the same for all uses (assuming you use the same exception type for all uses; if not, we'll see how we can fix that later).

We now have a template for dealing with file closing. Hmmm... I think there's a design pattern out there that might fit; in fact, it's named "Template Method".

The Template Method pattern is useful when you have an algorithm with replaceable steps. Some common steps of the algorithm always stay the same (and you usually want to enforce that they stay the same); some steps of the algorithm might need to be changed for different usage.

The "standard" implementation defines overridable "hook" methods for each variant step. The Template Method invokes these hook methods; subclasses can override the hooks to change the behavior of the algorithm.

Figure 7: File closing using a template method

public abstract class AutoFileCloser {

    // hook method; subclasses override to specify their work

    protected abstract void doWork() throws Throwable;

    private List<Closeable> streams = new ArrayList<Closeable>();

    protected final void watch(Closeable closeable) {

        streams.add(0, closeable);

    }

    // template method

    public final void go() throws SomeWrapperException {

        Throwable pending = null;

        try {

            doWork();

        } catch (ThreadDeath t) {

            throw t;

        } catch (Throwable t) {

            pending = t;

        } finally {

            for (Closeable c : streams) {

                try {

                    c.close();

                } catch (ThreadDeath t) {

                    throw t;

                } catch (Throwable t) {

                    // report and/or log error

                    if (pending == null)

                        pending = t;

                }

            }

            streams.clear();

            if (pending != null)

                if (pending instanceof SomeWrapperException)

                    throw (SomeWrapperException) pending;

                else

                    throw new SomeWrapperException(pending);

        }

    }

}

The go() method is final; this represents the constant parts of the Template Method that we do not want subclasses to be able to change. The watch() method  (which will be invoked in the subclass' doWork()) adds the streams to the front of the list. The ensures that streams will be closed in the opposite order in which they are opened.

With this definition in place, we can do the following:

Figure 8: Using the template method for some I/O

private static class MyWork extends AutoFileCloser {

    @Override protected void doWork() throws Throwable {

        FileReader r = new FileReader("someFile.txt");

        BufferedReader br = new BufferedReader(r);

        watch(r);

        watch(br);

        // process br

    }

}

new MyWork().go();

Now that's much better, but we still can make some improvements. If we make watch() a generic method, we can collapse the "new"s and the watch() calls:

    streams.add(0, closeable);

    return closeable;

}



// and then in the doWork()

When calling the AutoFileCloser, I prefer to use an anonymous inner class:

Figure 10: Calling AutoFileCloser as an anonymous inner class

new AutoFileCloser() {

Note: I like to collapse the closing }} together when I write anonymous inner classes. This makes it stand out: "something unusually ugly is happening here; maybe I should read more carefully!"

I'm not really keen on the dangling go() call. Let's move the go() contents to the AutoFileCloser constructor. The usage then looks like:

Figure 11: Moving the work to AutoFileCloser's constructor

new AutoFileCloser() {

    @Override protected void doWork() throws Throwable {

        FileReader r = watch(new FileReader("someFile.txt"));

        BufferedReader br = watch(new BufferedReader(r));

        // process br

    }};

Tada! We now have correct file closing that is not much more complex than what we started with at the beginning of this article!

This also works with multiple nested streams:

Figure 12: Nested streams in AutoFileCloser

new AutoFileCloser() {

    @Override protected void doWork() throws Throwable {

        FileReader r = watch(new FileReader("someFile.txt"));

        BufferedReader br = watch(new BufferedReader(r));

        FileWriter w = watch(new FileWriter("someOtherFile.txt"));

        BufferedWriter bw = watch(new BufferedWriter(w));

        // read from br and write to bw

    }};

Exception wrapping

There are several approaches we can take to wrap and throw exceptions so they can be handled outside the AutoFileCloser:

  • Wrap in RuntimeException. Many folks don't like this idea because it's normally used to represent what I call "programmer stupidity errors" (whoops! forgot that null check!).
  • Wrap in a custom unchecked exception (extend RuntimeException). This allows you to catch a specific exception for your project(s) but not need to have throws clauses throughout your call chains.
  • Wrap in a custom checked exception wrapper. This requires callers to handle the custom exception; you'll need to either have a try/catch block handle it or a throws clause in every method in your call chain

There is much debate on checked vs. unchecked exceptions; I tend to fall on the "unchecked" side of the debate, but I want this file-closing framework to work no matter which type of exception you prefer.

You may decide that you want to capture all exceptions thrown, not just the last pending exception.

Figure 13: Catching multiple exceptions

// An exception that tracks multiple causes

// If you change this to a checked exception, you'll need

//   to declare the AutoFileCloser constructor as throws MultiException

public class MultiException extends RuntimeException {

    private static final long serialVersionUID = 1L;

    private List<Throwable> causes;

    public MultiException(Throwable cause) {

        super(cause);

    }

    public MultiException(String message, Throwable cause) {

        super(message, cause);

    }

    public List<Throwable> getCauses() {

        if (causes == null)

            return Collections.singletonList(getCause());

        return Collections.unmodifiableList(causes);

    }

    public void addCause(Throwable newCause) {

        if (causes == null) {

            causes = new ArrayList<Throwable>();

            causes.add(getCause());

        }

        causes.add(newCause);

    }

    // you may want to also override printStackTrace to include

    //   the additional causes

}



public abstract class AutoFileCloser {

    protected abstract void doWork() throws Throwable;

    private List<Closeable> streams = new ArrayList<Closeable>();

    protected final <T extends Closeable> T watch(T closeable) {

        streams.add(0, closeable);

        return closeable;

    }

    public AutoFileCloser() {

        MultiException pending = null;

        try {

            doWork();

        } catch (ThreadDeath t) {

            throw t;

        } catch (Throwable t) {

            pending = new MultiException(t);

        } finally {

            for (Closeable closeable : streams) {

                try {

                    closeable.close();

                } catch (ThreadDeath t) {

                    throw t;

                } catch (Throwable t) {

                    if (pending == null)

                        pending = new MultiException(t);

                    else

                        pending.addCause(t);

                }

            }

            streams.clear();

            if (pending != null) {

                throw pending;

            }

        }

    }

}

To deal with that throws clause, we're going to need generics. We start by making the class generic:

Figure 15: Making AutoFileCloser generic

public abstract class AutoFileCloser<WrapperExceptionType extends Throwable> {

    public AutoFileCloser() throws WrapperExceptionType {

    ...

}

That takes care of the throws clause. Now we need to deal with the pending exception test-and-wrap. It would be nice if we could just say

Figure 16: Pending exception wrapping

// DOES NOT WORK IN JAVA...

if (pending != null)

    if (pending instanceof WrapperExceptionType)

        throw (WrapperExceptionType) pending;

    else

        throw new SomeWrapperException (pending);

This will not work in Java. Java erases the passed-in type of WrapperExceptionType; it's not available at runtime. (It's erased to Throwable, from the type parameter declaration).

We need an object that tells us the type of the exception. We can do this by passing a Class object into the AutoFileCloser constructor. We declare this as Class<WrapperExceptionType> so the user can only pass in the proper class.

Figure 17: Passing in the exception type

public AutoFileCloser(Class<WrapperExceptionType> wrapperExceptionType)

        throws WrapperExceptionType {

    ...

}

And now we can fixup that pending block a bit more

Figure 18: The pending block is getting a little better...

if (pending != null)

    if (wrapperExceptionType.isAssignableFrom(pending.getClass()))

        throw wrapperExceptionType.cast(pending);

    else

        throw new SomeWrapperException (pending);

One more location to fix! We add another hook method that's responsible for wrapping a Throwable. The subclass must implement this method to create the proper exception.

Figure 19: Asking the subclass to wrap an exception

protected abstract WrapperExceptionType wrap(Throwable t);

The final pending block is

Figure 20: The final pending exception block

if (pending != null)

    if (wrapperExceptionType.isAssignableFrom(pending.getClass()))

        throw wrapperExceptionType.cast(pending);

    else

        throw wrap(pending);

Returning a value

Things get a little trickier if you want to return a value from your doWork() method. I'll demonstrate two approaches:

  • You can declare a final variable in the same method in which you use AutoFileCloser, or a field in the class that acts as a handle to a "container" object (such as a Map or StringBuilder). These are accessible from within your doWork(). This is my preferred method of returning values from doWork().
  • You can modify AutoFileCloser to return a value (though you'll need to use a method such as "go" to return it).

Setting a return value via a container

This is a simple example of reading an entire file into a String. (It's actually part of the code I used to format the source code figures in this article. Yep - I'm eating my own dog food here!).

Figure 24: Setting a return value via a container

// example: fully reading a file into a StringBuilder

// The StringBuilder content is marked final, allowing it to be

//   accessed from within our anonymous inner class

final StringBuilder content = new StringBuilder();

new AutoFileCloser() {

    @Override protected void doWork() throws Throwable {

        FileReader fr = watch(new FileReader(file));

        char[] buf = new char[8192];

        int n;

        while ((n = fr.read(buf)) != -1)

            content.append(buf, 0, n);

    }};

Returning a value from AutoFileCloser

In this version, we modify AutoFileCloser to return a value. Because we cannot return a value from the constructor, this approach requires us to move the Template Method back to go() as we started with.

Figure 25: Returning a value from AutoFileCloser

public abstract class AutoFileCloser<ResultType> {

    ...

    // replace the constructor with go()

    public ResultType go() {

        Throwable pending = null;

        try {

            return doWork();

            ... // catches/finally et al

        }

        // can never get here, but needed to make compiler happy

        return null; 

    }

}



// and then using it

String content = new AutoFileCloser<String>() {

    @Override protected String doWork() throws Throwable {

        final StringBuilder content = new StringBuilder();

        FileReader fr = watch(new FileReader(file));

        char[] buf = new char[8192];

        int n;

        while ((n = fr.read(buf)) != -1)

            content.append(buf, 0, n);

        return content.toString();

    }}.go();

This is more complex than the final-variable approach, but this may prove useful under some circumstances. Note that if you define AutoFileCloser this way and don't have a value to return, you should pass Void as the ResultType, and your doWork() will need to return null.

Final Implementations

AutoFileCloser throwing RuntimeException

Here's the simplest version of the AutoFileCloser. You override it to define a doWork() method that performs your stream processing. The doWork() method can throw any type of exception. If an exception is thrown, it is wrapped in a RuntimeException to be handled by the caller or a caller further up the stack.

Figure 26: AutoFileCloser using RuntimeException as wrapper

import java.io.Closeable;

import java.io.FileReader;

import java.io.IOException;

import java.util.ArrayList;

import java.util.List;



public abstract class AutoFileCloser {

    protected abstract void doWork() throws Throwable;

    private List<Closeable> streams = new ArrayList<Closeable>();

    protected final <T extends Closeable> T watch(T closeable) {

        streams.add(0, closeable);

        return closeable;

    }

    public AutoFileCloser(){

        Throwable pending = null;

        try {

            doWork();

        } catch (ThreadDeath t) {

            throw t;

        } catch (Throwable t) {

            pending = t;

        } finally {

            for (Closeable c : streams) {

                try {

                    c.close();

                } catch (ThreadDeath t) {

                    throw t;

                } catch (Throwable t) {

                    // report and/or log error

                    if (pending == null)

                        pending = t;

                }

            }

            streams.clear();

            if (pending != null)

                if (pending instanceof RuntimeException)

                    throw (RuntimeException) pending;

                else

                    throw new RuntimeException(pending);

        }

    }

}



// example usage

new AutoFileCloser() {

    @Override protected void doWork() throws Throwable {

        FileReader fr = watch(new FileReader("somefile.txt"));

        // process fr

    }};

AutoFileCloser with Generic Exception

This version of AutoFileCloser is a little more complex. It can be used in different contexts that require different types of wrapper exceptions. These wrapper exceptions can be checked or unchecked exceptions. The subclass needs to implement the wrap() method to define how exceptions be wrapped.

Figure 27: AutoFileCloser with Generic Exception

import java.io.IOException;

import java.util.ArrayList;

import java.util.List;

    protected final <T extends Closeable> T watch(T closeable) {

        streams.add(0, closeable);

        return closeable;

    }

    protected abstract WrapperExceptionType wrap(Throwable t);

    public AutoFileCloser(Class<WrapperExceptionType> wrapperExceptionType)

            throws WrapperExceptionType {

        try {

            doWork();

AutoFileCloser with MultiException (and MultiException implementation)

This final implementation throws a MultiException, an exception that can track more than one exception that has been thrown.

Figure 28: AutoFileCloser throwing MultiException

// An exception that tracks multiple causes

// If you change this to a checked exception, you'll need

//   to declare the AutoFileCloser constructor as throws MultiException

public class MultiException extends RuntimeException {

    private static final long serialVersionUID = 1L;

    private List<Throwable> causes;

    public MultiException(Throwable cause) {

        super(cause);

    }

    public MultiException(String message, Throwable cause) {

        super(message, cause);

    }

    public List<Throwable> getCauses() {

        if (causes == null)

            return Collections.singletonList(getCause());

        return Collections.unmodifiableList(causes);

    }

    public void addCause(Throwable newCause) {

        if (causes == null) {

            causes = new ArrayList<Throwable>();

            causes.add(getCause());

        }

        causes.add(newCause);

    }

}



public abstract class AutoFileCloser {

    protected abstract void doWork() throws Throwable;

    private List<Closeable> streams = new ArrayList<Closeable>();

    protected final <T extends Closeable> T watch(T closeable) {

        streams.add(0, closeable);

        return closeable;

    }

    public AutoFileCloser() {

        MultiException pending = null;

        try {

            doWork();

        } catch (ThreadDeath t) {

            throw t;

        } catch (Throwable t) {

            pending = new MultiException(t);

        } finally {

            for (Closeable closeable : streams) {

                try {

                    closeable.close();

                } catch (ThreadDeath t) {

                    throw t;

                } catch (Throwable t) {

                    if (pending == null)

                        pending = new MultiException(t);

                    else

                        pending.addCause(t);

                }

            }

            streams.clear();

            if (pending != null) {

                throw pending;

            }

        }

    }

}
